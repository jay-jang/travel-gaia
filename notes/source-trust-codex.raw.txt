[검증전략] 조문/버전 환각을 막는 구체적 방법

1. `section`/`version`은 “검증 상태”가 아니라 “증거에서 추출된 값”으로만 취급한다.
   - WebSearch 결과만으로는 기입 금지.
   - 반드시 WebFetch로 1차 문서 본문, PDF 텍스트, 공식 HTML, 공식 schema/release note, 공식 resolution 목록 중 하나를 확인해야 한다.
   - 본문에서 조항명/번호/버전 문자열이 직접 확인되지 않으면 `section`/`version`은 비워 둔다.

2. 2단계 검증을 넣는다.
   - 1차 작성 에이전트: 후보 출처 교체 + `section/version` 후보 + 근거 스니펫 저장.
   - 검증 에이전트 또는 검증 스크립트: 같은 URL을 다시 fetch해서 `evidence_quote` 안의 문자열과 `section/version` 값이 실제 본문에 존재하는지 확인.
   - 사람이 보기 쉬운 diff에는 반드시 `before source`, `after source`, `evidence_quote`, `why_primary`를 남긴다.

3. `section`은 자유 입력을 허용하되 “근거 없는 정규화”를 금지한다.
   - 예: `Article 7`, `Art. 7`, `Resolution 830a`, `Category 31`, `NDC 21.3`처럼 문서에 나온 표기를 보존.
   - 에이전트가 “아마 Article 7일 것”이라고 판단한 값은 기입하지 않는다.
   - 용어 설명과 직접 연결되는 조항만 넣고, 문서 전체를 포괄 출처로 쓰는 경우 `section`은 비운다.

4. 못 찾으면 명시적으로 못 찾은 상태를 기록한다.
   - `verification_status: "primary_found_no_section"` 또는 `"primary_not_found"` 같은 상태를 둔다.
   - `notes`나 별도 audit 파일에 검색한 쿼리와 탈락 사유를 남긴다.
   - 빈 `section/version`은 실패가 아니라 정상 상태다. 허위 정밀도만 실패다.

5. 조항/버전 자동 보강은 allowlist 기반으로 한다.
   - EU261, Montreal Convention, IATA NDC, ACRISS, ATPCO, EMVCo, PCI DSS처럼 반복 등장하는 권위 문서만 문서별 패턴을 둔다.
   - 예: `eur-lex.europa.eu` + `Regulation (EC) No 261/2004`에서는 `Article N`만 허용.
   - IATA Resolution, ATPCO Category처럼 공개 접근이 제한되거나 요약 페이지가 많은 영역은 “공식 문서 본문 확인 실패 시 번호 기입 금지”.

[tier 분류] source tier 분류 체계

- `regulation`: 법령, 조약, 정부/초국가기구 규정 원문. 예: EUR-Lex, ICAO treaty text, national regulator.
- `standard-body`: 표준화 기구의 공식 표준, schema, rulebook, implementation guide. 예: IATA, EMVCo, ISO, PCI SSC.
- `association`: 산업 협회나 컨소시엄의 공식 정의/가이드. 예: ACRISS, GBTA, ASTA.
- `vendor-doc`: 특정 공급자/플랫폼의 공식 문서. 예: Amadeus, Sabre, Stripe, Booking.com partner docs.
- `operator-doc`: 항공사, 호텔, OTA, 보험사 등 서비스 운영 주체의 공식 정책/약관/도움말.
- `government`: 정부기관의 설명 페이지, 소비자 안내, FAQ. 법령 원문은 `regulation`, 해설은 `government`.
- `academic`: 논문, 대학, 연구기관 보고서.
- `press`: 보도자료, 공식 뉴스룸, 공시.
- `encyclopedia`: Wikipedia, Wikivoyage 등 범용 백과.
- `secondary`: 블로그, 컨설팅 글, SEO 설명글, 미디어 기사, 일반 가이드. 예: AltexSoft, Xotels, Squaremouth.
- `unknown`: URL 또는 조직 성격만으로 판정 불가.

[스키마] OKF source에 추가할 필드

- `tier`: 출처 권위 계층. 기존 `org/name/version/section/url` 옆에 optional로 추가.
- `primary`: 1차성 여부 boolean. `tier`와 별개로 빠른 집계용. 예: `regulation`, `standard-body`, `association`, 공식 `vendor-doc`는 보통 true.
- `verified_at`: 해당 URL과 조항을 확인한 날짜. ISO date.
- `verification_status`: `verified`, `primary_found_no_section`, `primary_not_found`, `paywalled`, `replaced_secondary`, `needs_review`.
- `evidence_quote`: WebFetch 본문에서 확인한 짧은 근거 문자열. 조항/버전 값을 넣은 경우 필수.
- `evidence_url`: 실제 확인 URL. 기존 `url`이 랜딩 페이지이고 PDF가 별도일 때 사용.
- `access`: `public`, `login_required`, `paywalled`, `archived`, `unknown`.
- `replaces`: 교체 전 출처의 간단한 fingerprint. 예: `{org,url}` 또는 source id.
- `confidence`: `high`, `medium`, `low`. 단, `section/version` 기입 가능 여부와 분리. 낮은 confidence면 조항 기입 금지.

후방호환은 기존 빌드가 모르는 source key를 무시하게 두면 된다. UI는 우선 `tier`와 기존 필드만 읽고, 검증용 필드는 빌드 산출물에 포함하되 표시하지 않아도 된다.

[게이트] okf-validate에 넣을 검증 규칙 3~5개

1. fail: `section` 또는 `version`이 있는 source는 `tier`, `verified_at`, `verification_status: verified`, `evidence_quote`가 모두 있어야 한다.
   - 목적: 조항/버전의 무근거 입력 차단.

2. fail: `section` 값이 있는데 `evidence_quote` 또는 fetched evidence cache 안에 `section` 문자열의 핵심 토큰이 없으면 실패.
   - 예: `Article 7`이면 `Article`/`Art.`와 `7` 동시 존재 필요.
   - 완벽한 NLP보다 “상상 번호 방지”에 초점.

3. warn: `tier: secondary|encyclopedia|unknown`인데 같은 엔트리의 모든 source가 이 tier뿐이면 경고.
   - 단일 2차 출처 엔트리를 계속 줄이기 위한 압박.

4. warn: `regulation|standard-body`인데 `section`이 비어 있으면 경고, 단 `verification_status: primary_found_no_section`이면 낮은 등급 경고.
   - “가능하면 조항까지 찾기”를 유도하되 허위 입력은 강제하지 않음.

5. fail: 동일 엔트리 안에서 같은 `url` + 같은 `section` + 같은 `version` source가 중복되면 실패.
   - 교체 스크립트 반복 적용으로 citations가 부풀어 오르는 것 방지.

[스킵 기준] 어떤 출처/용어는 손대지 말아야 하나

- 공식 문서가 유료, 로그인, 회원 전용이고 본문 확인이 안 되는 경우.

- 특정 벤더 기능명 또는 API 동작을 설명하는 항목.
  - 이 경우 표준기구보다 해당 벤더 공식 문서가 더 정확하다.
- Wikipedia/2차 출처가 역사적 배경이나 일반 설명을 담당하고, 1차 출처가 현재 정의만 제공하는 경우.
  - 교체가 아니라 보조 출처 추가가 맞다.
- 표제어 설명이 현재 OKF 내용과 1차 문서의 조항 범위가 다를 때.
  - 먼저 정의 텍스트를 고치지 않는 한 source만 바꾸면 citation drift가 생긴다.
- 폐지/구버전 표준을 설명하는 항목.
  - 최신 표준으로 덮어쓰면 오히려 오류다. `version`과 `deprecated` 성격을 명시해야 한다.

[리스크/함정] 이 작업의 최대 함정 2~3개

   - 용어 설명과 직접 연결되는 조항만 넣고, 문서 전체를 포괄 출처로 쓰는 경우 `section`은 비운다.

4. 못 찾으면 명시적으로 못 찾은 상태를 기록한다.
   - `verification_status: "primary_found_no_section"` 또는 `"primary_not_found"` 같은 상태를 둔다.
   - `notes`나 별도 audit 파일에 검색한 쿼리와 탈락 사유를 남긴다.
   - 빈 `section/version`은 실패가 아니라 정상 상태다. 허위 정밀도만 실패다.

5. 조항/버전 자동 보강은 allowlist 기반으로 한다.
   - EU261, Montreal Convention, IATA NDC, ACRISS, ATPCO, EMVCo, PCI DSS처럼 반복 등장하는 권위 문서만 문서별 패턴을 둔다.
   - 예: `eur-lex.europa.eu` + `Regulation (EC) No 261/2004`에서는 `Article N`만 허용.
   - IATA Resolution, ATPCO Category처럼 공개 접근이 제한되거나 요약 페이지가 많은 영역은 “공식 문서 본문 확인 실패 시 번호 기입 금지”.

[tier 분류] source tier 분류 체계

- `regulation`: 법령, 조약, 정부/초국가기구 규정 원문. 예: EUR-Lex, ICAO treaty text, national regulator.
- `standard-body`: 표준화 기구의 공식 표준, schema, rulebook, implementation guide. 예: IATA, EMVCo, ISO, PCI SSC.
- `association`: 산업 협회나 컨소시엄의 공식 정의/가이드. 예: ACRISS, GBTA, ASTA.
- `vendor-doc`: 특정 공급자/플랫폼의 공식 문서. 예: Amadeus, Sabre, Stripe, Booking.com partner docs.
- `operator-doc`: 항공사, 호텔, OTA, 보험사 등 서비스 운영 주체의 공식 정책/약관/도움말.
- `government`: 정부기관의 설명 페이지, 소비자 안내, FAQ. 법령 원문은 `regulation`, 해설은 `government`.
- `academic`: 논문, 대학, 연구기관 보고서.
- `press`: 보도자료, 공식 뉴스룸, 공시.
- `encyclopedia`: Wikipedia, Wikivoyage 등 범용 백과.
- `secondary`: 블로그, 컨설팅 글, SEO 설명글, 미디어 기사, 일반 가이드. 예: AltexSoft, Xotels, Squaremouth.
- `unknown`: URL 또는 조직 성격만으로 판정 불가.

[스키마] OKF source에 추가할 필드

- `tier`: 출처 권위 계층. 기존 `org/name/version/section/url` 옆에 optional로 추가.
- `primary`: 1차성 여부 boolean. `tier`와 별개로 빠른 집계용. 예: `regulation`, `standard-body`, `association`, 공식 `vendor-doc`는 보통 true.
- `verified_at`: 해당 URL과 조항을 확인한 날짜. ISO date.
- `verification_status`: `verified`, `primary_found_no_section`, `primary_not_found`, `paywalled`, `replaced_secondary`, `needs_review`.
- `evidence_quote`: WebFetch 본문에서 확인한 짧은 근거 문자열. 조항/버전 값을 넣은 경우 필수.
- `evidence_url`: 실제 확인 URL. 기존 `url`이 랜딩 페이지이고 PDF가 별도일 때 사용.
- `access`: `public`, `login_required`, `paywalled`, `archived`, `unknown`.
- `replaces`: 교체 전 출처의 간단한 fingerprint. 예: `{org,url}` 또는 source id.
- `confidence`: `high`, `medium`, `low`. 단, `section/version` 기입 가능 여부와 분리. 낮은 confidence면 조항 기입 금지.

후방호환은 기존 빌드가 모르는 source key를 무시하게 두면 된다. UI는 우선 `tier`와 기존 필드만 읽고, 검증용 필드는 빌드 산출물에 포함하되 표시하지 않아도 된다.

[게이트] okf-validate에 넣을 검증 규칙 3~5개

1. fail: `section` 또는 `version`이 있는 source는 `tier`, `verified_at`, `verification_status: verified`, `evidence_quote`가 모두 있어야 한다.
   - 목적: 조항/버전의 무근거 입력 차단.

2. fail: `section` 값이 있는데 `evidence_quote` 또는 fetched evidence cache 안에 `section` 문자열의 핵심 토큰이 없으면 실패.
   - 예: `Article 7`이면 `Article`/`Art.`와 `7` 동시 존재 필요.
   - 완벽한 NLP보다 “상상 번호 방지”에 초점.

3. warn: `tier: secondary|encyclopedia|unknown`인데 같은 엔트리의 모든 source가 이 tier뿐이면 경고.
   - 단일 2차 출처 엔트리를 계속 줄이기 위한 압박.

4. warn: `regulation|standard-body`인데 `section`이 비어 있으면 경고, 단 `verification_status: primary_found_no_section`이면 낮은 등급 경고.
   - “가능하면 조항까지 찾기”를 유도하되 허위 입력은 강제하지 않음.

5. fail: 동일 엔트리 안에서 같은 `url` + 같은 `section` + 같은 `version` source가 중복되면 실패.
   - 교체 스크립트 반복 적용으로 citations가 부풀어 오르는 것 방지.

[워크플로 설계] 카테고리 에이전트의 입력/출력/프롬프트 핵심 + 적용시 충돌/중복 처리

입력:
- 카테고리별 OKF 파일 목록.
- 현재 source 통계: secondary URL, missing URL, missing version/section, single-source entry.
- 도메인 우선순위 매핑.
  - `EU261` → `eur-lex.europa.eu`
  - `NDC`, `Resolution` → `iata.org`
  - `3DS` → `emvco.com`, 필요 시 `eba.europa.eu`
  - `ACRISS` → `acriss.org`
  - `ATPCO Cat` → `atpco.net` 공식 문서/교육 자료
  - `Montreal Convention` → ICAO/official treaty source

출력:
- JSONL patch manifest를 권장.
- 각 row:
  - `entry_path`
  - `term_id`
  - `action`: `replace_source`, `add_source`, `classify_only`, `leave_unchanged`
  - `old_source`
  - `new_source`
  - `evidence_quote`
  - `evidence_url`
  - `rationale`
  - `search_queries`
  - `review_required`

프롬프트 핵심:
- “조항/버전은 WebFetch 본문에 직접 나온 문자열만 입력한다.”
- “공식 문서가 paywall/login이면 조항 번호를 추정하지 말고 `paywalled`로 표시한다.”
- “2차 출처를 삭제하지 말고, 같은 주장에 대한 1차 출처가 확인될 때만 교체한다.”
- “용어 설명보다 넓은 문서 전체 출처라면 `section`을 비워 둔다.”
- “불확실하면 `review_required: true`.”

적용 스크립트:
- manifest를 읽어 frontmatter만 수정.
- 기존 source와 `new_source`를 canonical key로 비교:
  - canonical key = normalized URL host/path + `section` + `version`.
- 같은 canonical source가 있으면 merge:
  - 더 높은 tier 유지.
  - `evidence_quote`, `verified_at` 보강.
  - 기존 `name/org`가 더 사람이 읽기 좋으면 보존.
- source 순서는 권위도 기준:
  - `regulation`, `standard-body`, `association`, `vendor-doc`, `operator-doc`, `government`, `academic`, `press`, `secondary`, `encyclopedia`, `unknown`.
- 충돌 처리:
  - 같은 old source를 서로 다른 primary source로 교체하려 하면 자동 적용 금지.
  - 같은 URL에 서로 다른 version이 들어오면 둘 다 유지하지 말고 `review_required`.
  - 기존 사용자가 손댄 필드가 manifest 생성 이후 바뀌었으면 적용 스킵 후 리포트.

[스킵 기준] 어떤 출처/용어는 손대지 말아야 하나

- 공식 문서가 유료, 로그인, 회원 전용이고 본문 확인이 안 되는 경우.
  - IATA Resolution, ATPCO 일부 fare rule 문서가 여기에 걸릴 수 있다.
- 용어가 산업 속어, 관행, 마케팅 표현이라 표준 원문이 없는 경우.
  - 억지로 규정 출처를 붙이면 의미가 좁아진다.
- 특정 벤더 기능명 또는 API 동작을 설명하는 항목.
  - 이 경우 표준기구보다 해당 벤더 공식 문서가 더 정확하다.
- Wikipedia/2차 출처가 역사적 배경이나 일반 설명을 담당하고, 1차 출처가 현재 정의만 제공하는 경우.
  - 교체가 아니라 보조 출처 추가가 맞다.
- 표제어 설명이 현재 OKF 내용과 1차 문서의 조항 범위가 다를 때.
  - 먼저 정의 텍스트를 고치지 않는 한 source만 바꾸면 citation drift가 생긴다.
- 폐지/구버전 표준을 설명하는 항목.
  - 최신 표준으로 덮어쓰면 오히려 오류다. `version`과 `deprecated` 성격을 명시해야 한다.

[리스크/함정] 이 작업의 최대 함정 2~3개

1. 공개 웹에 있는 “공식처럼 보이는 요약 페이지”를 원문으로 착각하는 것.
   - IATA, ATPCO, EMVCo는 소개 페이지와 실제 표준 문서의 신뢰 수준이 다르다.
   - 요약 페이지는 `standard-body`일 수는 있어도 `section/version` 근거가 아닐 수 있다.

2. 조항 단위 출처와 용어 정의의 범위를 맞추지 못하는 것.
   - 예: EU261 `Article 7`은 compensation amount에는 맞지만 “EU261 전체” 설명에는 너무 좁다.
   - section 정밀화는 citation 품질을 올리지만, 잘못 붙이면 설명 범위를 왜곡한다.

3. 자동화가 “빈칸 채우기 게임”이 되는 것.
   - version/section coverage를 KPI로 잡으면 환각 압력이 생긴다.
   - KPI는 “verified section count”와 함께 “explicitly left blank with reason”도 같이 봐야 한다.

[한줄 총평] 가장 중요한 한 가지

`section/version`은 채워야 하는 메타데이터가 아니라, WebFetch로 본문에서 직접 검출된 증거의 부산물로만 저장해야 한다.
