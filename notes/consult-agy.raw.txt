### 1. 도메인 커버리지 피드백 (빠진 것 / 합칠 것 / 수정할 것)
*   **추가(빠진 것)**
    *   **Identity & Loyalty**: `IATA One ID`(생체 인식 기반 신원 인증), passenger type codes(ADT/CHD/INF), FQTV(Frequent Flyer Number).
    *   **Payment & B2B Settlement**: OTA-호텔/GDS 정산의 핵심인 `VCC(Virtual Credit Card)`, `3DS` 보안 인증, IATA의 새로운 결제 표준 `IATA EasyPay`.
    *   **Hotel Protocols**: HTNG(Hotel Technology Next Generation) 표준 스펙, `ARI(Availability, Rate, Inventory)` Push/Pull 전송 모델.
*   **통합 권고**
    *   `Air-Reservation`, `Air-Interlining`, `Air-Codes`는 개념적 중복이 많습니다. 예를 들어 `leg/segment`가 양쪽에 흩어져 있고, PNR과 SSR/OSI는 결합도가 높습니다. 이를 `Air-Operations & Reservation` 카테고리로 묶어 단순화하는 것을 추천합니다.
*   **수정 권고**
    *   `EMD`는 `Air-Ticketing`에, `EMD-A`는 `Ancillary/Other`에 흩어져 있습니다. EMD-A(Associated)와 EMD-S(Standalone)는 하나의 `EMD/Ancillary` 개념 하위로 모아 통합 기술해야 일관성이 있습니다.

### 2. "용어 통일" 데이터 모델 개선점
*   **동의어/충돌 명칭의 구조화**
    *   단순 1:1 매핑(`"Amadeus": "TST"`)은 공급사별 동작 맥락의 미세한 차이를 담지 못합니다. 명칭과 컨텍스트(설명)를 함께 갖는 객체 구조로 개선할 것을 권장합니다.
    ```json
    "providerTerms": {
      "Amadeus": { "term": "TST", "context": "Transitional Store Ticket. 발권 전 운임 저장소" },
      "Sabre": { "term": "PQ", "context": "Price Quote. Amadeus의 TST와 기능적 유사" }
    }
    ```
*   **표준 기구 정의 명시**
    *   해당 용어가 공인 표준(`IATA`, `HTNG`, `OpenTravel`)인지 개별 벤더 독자 규격인지 구분할 수 있는 `standardBody` 필드를 추가합니다.
*   **계층 및 관계 방향 정의**
    *   단순 `related` 외에 `parent`(상위 개념), `children`(하위 개념), `conflicts`(동음이의어/충돌 표기)를 구분하여, 항공의 Segment와 호텔의 Segment처럼 도메인에 따라 의미가 달라지는 문제를 방지합니다.

### 3. Standalone 단일 HTML 접근의 함정과 권고사항
*   **함정 1: 데이터 크기 팽창 및 파싱 지연**
    *   *분석*: 상세 설명과 스펙 소스가 늘어날수록 JSON 임베딩 용량이 급증하여 초기 로딩과 DOM 렌더링 성능이 저하됩니다.
    *   *권고*: 바닐라 JS와 CSS `content-visibility: auto`를 사용하여 렌더링 성능을 최적화하고, 빌드 타임에 미니파이(Minify)를 강제합니다.
*   **함정 2: 클라이언트 사이드 검색의 한계**
    *   *분석*: 단순 단어 포함 여부(`indexOf`) 검색은 오타, GDS별 동의어 매핑 검색을 지원하지 못해 검색 UX가 떨어집니다.
    *   *권고*: [MiniSearch](https://github.com/lucaong/minisearch)나 FlexSearch 같은 초경량(수 KB 수준) JS 검색 라이브러리를 인라인 임베딩하여 Fuzzy Search 및 가중치 필드 검색을 지원합니다.
*   **함정 3: 공유 및 북마크 불가능 (Deep Linking)**
    *   *분석*: 로컬 단일 파일(`file:///...`)에서는 URL 경로 분기가 불가능해 특정 용어 페이지를 다른 사람에게 직접 공유하기 어렵습니다.
    *   *권고*: Hash 라우터(`window.addEventListener('hashchange')`) 기반의 `index.html#/term/pnr` 주소 규칙을 설계하여, 실행 시 특정 용어를 바로 팝업하거나 포커싱하도록 만듭니다.

### 4. 반드시 포함해야 할 핵심 표제어 20선
1.  **PNR (Passenger Name Record)**: 여행 예약 정보 기록의 기본 단위.
2.  **GDS (Global Distribution System)**: 항공/호텔 유통을 매개하는 전통적인 글로벌 분배 시스템.
3.  **NDC (New Distribution Capability)**: XML/JSON 기반의 IATA 차세대 항공권 유통 규격.
4.  **Offer & Order**: NDC 및 ONE Order 환경에서 PNR과 e-Ticket을 대체하는 주문 중심 모델.
5.  **RBD (Reservation Booking Designator)**: 예약을 제어하는 예약 등급 알파벳 코드.
6.  **Fare Basis Code**: 요금 규정과 할인 조건을 식별하는 영문·숫자 조합 코드.
7.  **EMD (Electronic Miscellaneous Document)**: 부가 서비스(수하물, 좌석 지정 등) 판매/발권 표준.
8.  **Interline Agreement**: 복수 항공사 구간 결합 및 연계 수송을 가능케 하는 다자간 협약.
9.  **Codeshare**: 공동운항. 실제 운항사(Operating)와 판매사(Marketing)의 권리/의무 분리.
10. **BSP (Billing and Settlement Plan)**: IATA 대리점과 항공사 간 대금 정산 시스템.
11. **ARC (Area Settlement Plan / Airline Tariff Publishing Company)**: 미주 정산(ARC) 및 전 세계 항공 운임 등록/배포 기관(ATPCO).
12. **MCT (Minimum Connection Time)**: 여정 생성 및 발권 시 준수해야 하는 공항별 최소 연결 시간.
13. **ARNK (Arrival Unknown)**: 여정 중 항공편이 아닌 다른 수단으로 이동하여 단절된 세그먼트.
14. **OTA (Online Travel Agency) / OpenTravel**: 온라인 여행사 구조 및 XML 스키마 데이터 교환 표준 기구.
15. **ARI (Availability, Rate, and Inventory)**: 호텔 공급망에서 핵심 예약 상태 정보를 전송하는 데이터 규격.
16. **Channel Manager**: 여러 판매 채널의 객실 수량과 요금을 실시간 동기화하는 미들웨어.
17. **ADR & RevPAR**: 호텔 비즈니스의 핵심 실적 평가 지표 (Average Daily Rate, Revenue Per Available Room).
18. **BAR (Best Available Rate)**: 실시간 수요에 따라 변동하는 호텔의 당일 최적 운임 정책.
19. **VCC (Virtual Credit Card)**: OTA와 호텔/공급사 간 안전한 대금 지불을 위해 생성되는 일회성 가상 신용카드.
20. **Passive Segment**: 실제 좌석을 잡지 않고 타사 예약 확인이나 발권 목적으로 GDS에 강제 입력하는 세그먼트.

### 5. LLM 환각(Hallucination) 방지 및 데이터 정확성 확보 방안
*   **소스 데이터 분리 및 빌드 파이프라인 구축**
    *   개발 시 단일 HTML을 직접 수정하지 말고, `data/` 디렉토리에 카테고리별 YAML/JSON 소스 파일을 분할하여 관리합니다.
    *   빌드 스크립트(Python/Node.js)를 통해 정적 유효성 검사(Schema Validation)를 완료한 후 단일 `index.html`로 컴파일하는 방식을 채택합니다.
*   **엄격한 출처(Sources) 스키마 정의**
    *   모든 표제어에 IATA Resolution 번호, NDC Specification 스펙 버전 번호, ATPCO 규정집 번호 등을 필수로 기록하도록 스키마 수준에서 강제하고, UI 화면에 이를 명시적으로 노출합니다.
*   **코드 검증 도구 활용**
    *   공항 IATA/ICAO 코드 리스트, SSR 코드 가이드라인 등 표준 규격을 정적 JSON 파일 형태로 검증 스크립트에 탑재해 빌드 단계에서 실시간 대조 검증을 수행합니다.
