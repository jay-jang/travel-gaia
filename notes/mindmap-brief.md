# Travel Wiki — 마인드맵 UI 설계 자문

너는 시니어 데이터비주얼라이제이션 엔지니어다. 아래 기능의 **설계**를 제안하라. 결론 지향, 구체적으로.

## 제품/현황
Travel Wiki = 여행산업 용어 백과사전. **407개 표제어**, 명확한 계층: **vertical(5) → subcategory(16) → term(407)**. 각 term엔 relationships/distinctions(교차 링크)도 있음.
- 단일 `index.html`, 의존성 0, 오프라인. **d3 v7 인라인 임베드**됨(이미 사용 중). KO/EN 토글, 다크모드.
- 기존 뷰: 편집형 표제어 페이지, **D3 force-directed 관계 그래프**(카테고리→부모→자식, 관계 기반·확장형), 전체 색인(A-Z), 비교/공급자매트릭스, vertical 랜딩 페이지.

## 추가하려는 것
**마인드맵 UI** — 전체 용어를 **계층적으로 구조화해 한눈에 이해**하도록. 기존 관계 force-graph와는 **다른 목적**(관계 탐색이 아니라 **전체 분류 구조의 조망/이해**). 라우트 예: `#/mindmap`.

## 제약
- 단일 HTML·오프라인·d3 v7 사용. 기존 그래프/클래스룸 코드 **건드리지 않고 additive**. KO/EN.
- 407개 리프를 한 번에 그리면 과밀 → 점진적 공개 필요.

## 질문 (이대로 답하라)
```
[레이아웃] 최적 마인드맵 레이아웃 1개 추천 + 이유 (radial tree / dendrogram / collapsible indent tree / treemap / circle-packing 중). 대안 1개.
[407 처리] 노드 과밀을 푸는 방식 (기본 vertical/category만, 클릭 확장, 검색 점프, 줌/팬 등)
[상호작용] 노드 클릭/호버 동작, 표제어로의 네비, 접기/펼치기, 검색 연동
[force-graph와 차별] 기존 관계 그래프와 무엇이 다르고 어떻게 공존하나
[관계 표현] 계층(트리) 외에 cross-link(relationships/distinctions)를 마인드맵에 어떻게(또는 안 넣을지)
[구현] d3 v7로 단일파일 내 구현 핵심(어떤 d3 모듈/레이아웃, SVG vs canvas, 반응형, 다크모드/KO-EN)
[모바일] 좁은 화면 대응
[리스크] 최대 함정 2~3개
[한줄총평] 가장 중요한 한 가지
```
일반론 말고 **이 제품(계층 5→16→407, d3 인라인, 단일파일)에 특정된** 구체안으로.
